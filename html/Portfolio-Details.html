• 프로젝트 처음 진행 당시 개발에 대한 아무런 지식이 없었기에 예측까지는 성공하였으나

배운 내용
1. GeoJson 사용방법
• GeoJson 시각화 시에 특정 모양으로 시각화 하려면 Json 파일이 필요하다는 것을 배웠습니다.
• 지도 시각화와 범례 사용에 대해 자세히 배울 수 있었습니다. 지도 시각화는 몇 배율로 표시하는지가 중요한데 이를 범례를 통해 더욱 시각적으로 표시하는 방법을 배웠습니다.

2. 데이터 전처리 및 결합
• 데이터를 전처리 시에 pandas를 통해 더욱 효율적으로 전처리하는 방법을 배웠습니다.

3. 객관적인 분석의 중요성
• 이 대회를 진행할 당시 단순한 분석 및 시각화로 그쳤으며, 이를 팀원들과 함께 임의로
분석하였습니다. 이로 인해 좋지 못한 성적을 거두었으며, 이를 계기로 데이터를 분석할 때,
더욱 객관적인 과정 및 결과가 필요하다는 것을 크게 배웠습니다.
문제점
1. 데이터 키(지역 코드) 불일치 문제
• 지도 시각화에 익숙치 않아 데이터키와 지역을 제대로 매칭시키지 못하였고 이로 인해 특정 지역에 대한 시각화가 제대로 진행되지 않는 문제가 발생하였습니다.

2. 지역 누락 문제
• Json 파일을 처음부터 만드는 것은 불가능하다고 판단하여 온라인에서 기존의 파일을 찾아
수정하여 사용하였습니다.
그러다보니 제가 분석한 지역과는 매칭이 되지 않았고, 이로 인해 특정 지역에 대한 시각화가
진행되지 않는 문제가 발생하였습니다.
해결법
1. 데이터 키 수정: 엑셀의 데이터 키를 Json 파일에 맞게 수정하여 데이터키 불일치 문제를
해결하였습니다.

2. 새로운 Json 파일 탐색: 온라인에서 전국 8도로 나누어진 Json 파일을 찾아 새로운 Json 파일로 설정, 지역 누락 문제를 해결하였습니다. 물론 Json 파일 없이 코드 자체에서 해결할 수 있었지만
저희는 전국 8도라는 특정한 모양의 지도 시각화를 원하였고, Json 파일을 찾는 것이 더욱
효율적이라는 판단하에 시행하였습나다.

• 학습 데이터와 테스트 데이터의 비율을 잘못 설정할 경우 과적합으로 인한 예측 오류가

2. Stadnard_Scaling
• 데이터의 스케일을 조정하는 것이 모델 성능에 미치는 영향을 학습하였습니다.
• 데이터의 부족으로 인해 예측의 부족함을 스케일링을 통해
조금이나마 개선할 수 있음을 배웠습니다.
• 데이터 스케일링 이후, 여러 모델을 테스트 한다면 성능을 크게 향상시킬 수 있음을 배웠습니다.
문제점
1. 데이터 과적합 및 특정 지역의 데이터에 대한 예측치가 너무 낮은 문제 • 모델이 특정 데이터에
과도하게 학습되면서 과적합 현상이 발생하였다.
• 일부 지역의 데이터가 제대로 반영되지 않아 예측값이 낮게 나오는 문제가 발생하였다.
해결법
1. Standard_Scaling을 통한 데이터 과적합 완화
• 데이터 스케일링을 통해 데이터 분포를 일정하게 조정하였으며, 모델이 특정 변수에 과도하게
의존하지 않도록 하였습니다.
• 각 데이터의 분포도를 데이터 스케일링을 통해 조절함으로써, 변수의 예측치를 높일 수 있었습니다.

2. 여러 모델을 통한 비교 후, 가장 예측에 대한 R2 Score가 높은 모델을 선정
• 데이터 스케일링과 Train_test_split으로 학습시킨 데이터를 여러 모델틀 통해 비교하였습니다.
• 이후, 각 모델의 예측치 결과 R2 Score가 가장 높은 모델을 선정하였습니다.

• 장바구니, 주문, 결제 확인, 배송정보 순서로 참조해야 하지만 역방향으로 참조하여
정상적인 로직으로 하면 DB가 생성되지 않았습니다.
해결법
1. 간편결제 제거
• 처음 설계가 카카오페이, 네이버페이 두 가지를 모두 적용하려는 것이었습니다.
• 둘 중, 하나만 적용하는 것은 의미가 없다고 판단하였으며, 카카오페이와 네이버페이 모두
KG 이니시스 안에 있어 간편 결제 자체를 제거하는 것으로 설계를 변경하였습니다.

2. 데이터 역방향 생성
• DB를 다시 정방향으로 다시 설계하기에는 시간이 부족하다고 판단하였습니다.
• 문제가 발생하는 부분은 결제 성공 시, 생성되는 데이터의 문제였기에, 데이터 생성을
역방향으로 생성되게 하여 문제를 해결하였습니다.

배운 내용
1. 웹소켓의 개념
• 클라이언트와 서버 간의 실시간 양방향 통신을 지원하는 프로토콜
• 기존 HTTP 요청/응답 방식과 달리, 한 번 연결을 맺으면 지속적으로 데이터를
주고받을 수 있습니다.
• 실시간 채팅, 게임, 주식 거래 등 빠른 데이터 전송이 필요한 서비스에서 많이 사용됩니다.
• 소켓을 유지하며 데이터를 이벤트 기반으로 주고받습니다.
• 클라이언트와 서버가 연결을 유지하면서 필요할 때만 데이터를 주고받아
네트워크 부하가 줄어듭니다.
• HTTP와 달리 요청을 보내지 않아도 서버에서 클라이언트로 데이터를 푸시할 수 있습니다.

2. 웹소켓과 HTTP/기타 프로토콜과의 차이점
• HTTP는 요청과 응답이 1:1 방식으로 연결이 끊기는 반면, 웹소켓은 한 번 연결되면 유지됩니다.
• 웹소켓은 Handshake과정 이후 지속적인 연결을 유지하며
데이터를 주고받을 수 있습니다.
• HTTP는 요청마다 헤더가 포함되어 오버헤드가 크지만, 웹소켓은 프레임 기반 통신으로
데이터 전송이 가볍습니다.
• 지속적인 연결을 유지하기 때문에 실시간성이 중요한 서비스에서 웹소켓이 더 적합합니다.

3. 웹소켓을 활용한 실시간 통신 구현
• HTTP 요청과 달리 웹소켓은 지속적인 연결을 유지하여 실시간 데이터 전송이 가능하다.
• 클라이언트와 서버 간의 양방향 통신을 통해 효율적인 메시지 전송이 이루어진다.
문제점
1. 소켓 ID 중복 처리 문제
• 만약 다른 기기에서 접속 중인 사용자라면, 새로 입력된 소켓 ID로 덮어씌워져
이전에 연결된 소켓 ID와 연결된 상태 정보가 덮어씌워져 문제가 발생하였습니다.

2. 현재 대화 상대가 누구인지 모르는 문제
• 채팅방을 열었지만, 상대방이 누구인지 표시되지 않았습니다.
해결법
1. 사용자가 여러 기기에서 접속할 수 있도록 처리
• 코드에서 user.token = sid;로 소켓 ID를 새로 갱신하고 있습니다.
• 이 때, 새로운 소켓 ID가 들어와도 기존 소켓 ID를 갱신하되, 다른 장치에서 접속한 상태를
명시적으로 처리하는 방식으로 문제를 해결하였습니다.

2. 채팅방 진입 시 상대방 정보 로드
• 채팅방을 열 때 프로프트로 이름을 입력하게 하였습니다.
• 클라이언트에서 상대방의 닉네임을 채팅방 상단에 표시하여, 사용자가 누구와 대화 중인지
알 수 있도록 하여 문제를 해결하였습니다.

배운 내용
1. S3를 통한 사진 입출력
• S3 버킷에 파일을 업로드하고, 업로드된 파일을 다운로드하는 방법을 배웠습니다.
이를 통해 클라우드 저장소에 파일을 안전하게 보관할 수 있음을 이해했습니다.
• 버킷에 대한 접근 제어와 권한을 설정하여, 필요한 파일만 외부에서 접근할 수 있게 하는
방법을 배웠습니다.
• 업로드된 파일에 접근할 수 있는 URL을 생성하고, 이를 이용해 웹 애플리케이션에서 사진을
표시하는 방법을 익혔습니다.

2. 카카오 지도와 마커 표시
• 카카오 지도를 웹페이지에 삽입하고, 지도에 마커를 추가하는 방법을 배웠습니다.
이를 통해 사용자 위치나 중요한 장소를 지도에 표시할 수 있게 되었습니다.
• 사용자 또는 서버에서 받아온 데이터를 기반으로 동적으로 마커를 지도에 표시하는
방법을 배웠습니다.
• 마커 클릭 등 지도에서 발생하는 다양한 이벤트를 처리하여, 사용자 인터랙션을 보다
풍부하게 만드는 방법을 익혔습니다.

3. 파이어베이스를 통한 로그인
• 파이어베이스에서 인증 기능을 설정하고, 이메일/비밀번호 기반 로그인 방식을
구현하는 방법을 배웠습니다.
문제점
1. S3를 통한 사진 출력
• S3를 통해 사진 업로드는 문제 없었는데, 출력시 출력이 되지 않는 문제가 발생하였습니다.
• 확인 결과, 출력시 접근하는 주소체계에 문제가 있었습니다.

2. 로그인 인터셉터
• 로컬에서는 화이트리스트 방식으로 잘 작동하였슨데, 배포시에는 제외시키는 코드가
작동하지 않았습니다.
• 이로 인해 배포 시, 무한리다이렉션이 발생하였습니다.

3. 경로 관련 에러
• 배포 시, 분명 html 파일이 있음에도 불구하고 404에러가 발생하였습니다.

해결법
1. 주소 컨버터 로직 삽입
• 확인 결과, 버킷에서 직접 접근하는 주소와 프로젝트 코드 내에서 접근하는 주소에
순서가 다름을 확인하였습니다.
• 버킷에서 직접 접근하는 주소로는 접근이 가능하였습니다.
• 프로젝트에서 접근하는 주소의 순서만 바꾸면 버킷에서 접근 하는 주소로 변환이 가능하며,
이를 실행하는 컨버터 로직을 삽입하였습니다.

2. 블랙 리스트 방식으로 대체
• 화이트 리스트 방식을 포기하고, 필요 페이지를 막는 블랙리스트 방식을 통해
문제를 해결하였습니다.

3. "/"제거
• return 앞에 "/"를 제거하여 문제를 해결하였습니다.
• 로컬에서는 프로그램이 경로를 보정해주지만, 배포시에는 그렇지 못하여 발생한 문제였습니다.
