문제 1. 지역 코드 불일치로 인한 지도 시각화 오류
문제 상황
• 지도 시각화 결과 일부 지역만 표시되지 않는 현상을 발견했습니다. 분석 데이터는 정상적으로 존재했기 때문에 데이터 누락이 아닌 지도와 데이터 간 매칭 문제라고 판단했습니다.
해결 과정
• 원인을 파악하기 위해 GeoJson 파일과 엑셀 데이터의 구조를 비교했습니다. 그 결과, 엑셀 데이터의 지역 코드와 GeoJson 파일의 지역 키 형식이 서로 다르다는 점을 확인했습니다.

• 해결 방안으로는 GeoJson 파일의 키 구조를 수정하는 방법과 엑셀 데이터의 지역 코드를 GeoJson 형식에 맞게 수정하는 방법 두 가지를 고려했습니다.

• GeoJson 파일은 지도 전체 구조에 직접적인 영향을 주는 핵심 데이터였기 때문에, 구조 변경 시 발생할 수 있는 시각화 오류를 최소화하고자 엑셀 데이터의 지역 코드를 수정하는 방법을 선택했습니다.
결과 및 개선
• 데이터 키가 정상적으로 매칭되면서 누락되었던 지역이 지도에 정상적으로 표시되었고, 추가적인 시각화 오류 없이 문제를 해결할 수 있었습니다.
문제 2. GeoJson 지역 누락으로 인한 시각화 미표시 문제
문제 상황
• 기존 GeoJson 파일을 수정하여 사용하던 중, 분석 대상 지역의 데이터는 존재했지만 지도에는 해당 지역이 표시되지 않는 현상을 발견했습니다. 이로 인해 GeoJson 데이터 자체에 지역 누락이 발생했음을 인지했습니다.
해결 과정
• 문제 해결을 위해 두 가지 방법을 검토했습니다. 첫째, 기존 GeoJson 파일에 누락된 지역을 직접 추가하는 방법과 둘째, 분석 대상과 구조가 일치하는 새로운 GeoJson 파일을 적용하는 방법이었습니다.

• 기존 파일을 직접 수정하는 경우, 좌표 오류나 지도 형태 왜곡이 발생할 가능성이 있다고 판단했습니다. 이에 따라 전국을 8도로 구분한 GeoJson 파일을 새로 탐색하여 적용했고, 분석 데이터와 정확히 일치하는 구조로 변경하는 방식을 선택했습니다.
결과
• 누락되었던 지역이 지도에 정상적으로 표시되었으며, 의도한 전국 8도 형태의 지도 시각화를 안정적으로 구현할 수 있었습니다.

• 이를 통해 결제 시스템에서 데이터 무결성을 유지하는 것이 얼마나 중요한지 깨달았습니다.
문제. 배송 테이블 역참조 설계로 인한 데이터 생성 충돌 문제
상황
• 주문, 배송, 결제 정보를 분리하여 테이블을 설계하였으나, 참조 방향을 잘못 설정하는 실수를 하였습니다.
• 정상적인 구조라면 배송 테이블이 주문을 참조(FK)해야 하지만, 초기 설계에서는 주문 테이블이 배송을 참조하는 구조로 설계되어 있었습니다.
• 이로 인해 주문을 생성하려면 먼저 배송 데이터가 존재해야 했고, 배송 데이터를 생성하려면 주문 ID가 필요한 구조가 되어 데이터 생성 순서가 논리적으로 충돌하는 문제가 발생하였습니다.
• 특히 결제 성공 이후 주문 및 배송 정보를 생성하는 과정에서 FK 제약 조건 오류가 발생하거나 insert 자체가 실패하는 상황이 반복되었습니다.

실행
• 해결 방법은 두 가지였습니다. 첫째, DB를 전면 재설계하여 FK 방향을 정상 구조로 수정하는 방법. 둘째, 기존 구조를 유지한 채 데이터 생성 순서를 조정하는 방법이었습니다.
• 전면 재설계가 가장 이상적이었으나, 이미 Controller → Service → DAO 전반에 걸쳐 해당 구조를 기반으로 로직이 작성되어 있었기 때문에 수정 범위가 매우 광범위했습니다.

• 프로젝트 일정과 안정성을 고려하여 구조는 유지하되 데이터 생성 순서를 조정하는 방식을 선택하였습니다.

• 리액트에서는 화면을 여러 개의 작은 컴포넌트로 나누어서 만든다는 것을 배웠습니다. 예를 들어, 입력창, 버튼, 목록을 각각 독립적인 컴포넌트로 만들 수 있어요
• 이를 통해, 코드를 더 깔끔하고 보기 쉽게 정리할 수 있으며, 같은 컴포넌트를 여러 번 재사용할 수 있어서 유지보수가 편리하다는 것을 배웠습니다.

2. 상태 관리에 대해 배웠습니다.
• 상태(State) 는 컴포넌트가 가지고 있는 값(데이터)을 의미하다는 것을 배웠습나다.
• 예를 들어, 할 일 목록을 화면에 보여주려면, 할 일 목록을 상태로 저장해야 하며, 리액트에서는 useState라는 기능을 사용해서 상태를 관리한다는 것을 배웠습니다.
• 이것이 중요한 이유는 상태가 바뀌면 자동으로 화면이 업데이트되기 때문에, 직접 DOM을 조작할 필요가 없기 때문입니다.
문제. React 상태 관리 개념 미숙으로 인한 할 일 목록 갱신 실패
문제 상황
• React를 처음 사용한 프로젝트에서 입력값을 추가해도 화면에 새로운 할 일이 렌더링되지 않는 문제가 발생하였습니다.
• 일반 변수에 값을 저장하는 방식으로는 컴포넌트가 재렌더링되지 않는다는 점을 인지하지 못한 상태였습니다.

실행 과정
• React는 상태(State)가 변경될 때만 컴포넌트가 재렌더링된다는 구조를 학습하였습니다.
• 이에 따라 const [inputValue, setInputValue] = useState('')와 const [todoList, setTodoList] = useState([])를 선언하여 입력값과 목록을 상태로 관리하도록 수정하였습니다.
• addItem 함수에서 setTodoList([...todoList, inputValue]) 방식으로 기존 배열을 직접 수정하지 않고, 새로운 배열을 생성하여 상태를 업데이트함으로써 React의 불변성 원칙을 따르도록 구현하였습니다.
• 또한 setInputValue('')를 통해 상태를 초기화함으로써 입력 필드가 자동으로 비워지도록 UI와 상태를 일치시켰습니다.
• 이 과정에서 useState는 “값을 저장하고, 그 값의 변경을 통해 렌더링을 유도하는 역할”이라는 점을 명확히 이해하게 되었습니다.
• 한편, useEffect는 상태 변화 이후 특정 로직을 실행하는 ‘부수 효과(side effect)’를 처리하기 위한 훅으로, 데이터 저장이나 API 호출, 구독 처리 등에 적합하다는 점을 학습하였습니다.
• 본 기능은 단순히 입력값을 저장하고 목록을 갱신하는 구조였기 때문에, 부수 효과 처리가 아닌 상태 자체의 변경이 핵심이었으며 useEffect를 사용할 필요는 없다고 판단하였습니다.
• 또한 React.memo는 불필요한 재렌더링을 방지하기 위한 최적화 도구로, 상태 변경을 직접 다루는 훅이 아니기 때문에 현재 문제 해결과는 목적이 다르다는 점을 구분하였습니다.
• 이를 통해 useState는 “상태 관리”, useEffect는 “사이드 이펙트 제어”, memo는 “렌더링 최적화”라는 각 훅의 역할 차이를 명확히 구분하게 되었습니다.

결과
• 입력값과 할 일 목록이 정상적으로 상태와 연결되며, 버튼 클릭 시 즉시 화면이 갱신되도록 개선하였습니다.
• React의 선언형 렌더링 구조와 상태 중심 설계 방식에 대한 이해도를 높이는 계기가 되었습니다.

• 클라이언트와 서버 간의 실시간 양방향 통신을 지원하는 프로토콜

2. 웹소켓과 HTTP/기타 프로토콜과의 차이점
• HTTP는 요청과 응답이 1:1 방식으로 연결이 끊기는 반면, 웹소켓은 한 번 연결되면 유지됩니다.
• 웹소켓은 Handshake과정 이후 지속적인 연결을 유지하며 데이터를 주고받을 수 있습니다.
• HTTP는 요청마다 헤더가 포함되어 오버헤드가 크지만, 웹소켓은 프레임 기반 통신으로 데이터 전송이 가볍습니다.
• 지속적인 연결을 유지하기 때문에 실시간성이 중요한 서비스에서 웹소켓이 더 적합합니다.

3. 웹소켓을 활용한 실시간 통신 구현
• HTTP 요청과 달리 웹소켓은 지속적인 연결을 유지하여 실시간 데이터 전송이 가능하다.

문제. 다중 기기 접속 시 소켓 ID 관리 문제
문제 상황
• 사용자가 여러 기기에서 동시에 접속할 경우, 새로운 소켓 ID가 기존 ID를 덮어씌우면서 이전 연결의 상태 정보가 손실되는 문제가 발생하였습니다.
• 그 결과 특정 기기에서는 메시지를 수신하지 못하거나, 연결이 끊긴 것으로 처리되는 현상이 발생하였습니다.

실행 과정
• 기존 구조에서는 user.token = sid; 형태로 하나의 사용자에 하나의 소켓 ID만 저장하고 있었습니다.
• 이 구조는 단일 기기 접속만 고려한 설계였기 때문에, 다중 접속 환경에서는 상태 관리에 문제가 발생한다는 점을 분석하였습니다.
• 따라서 사용자 객체에 단일 소켓 ID를 저장하는 방식이 아닌, 소켓 ID를 배열 형태로 관리하는 구조로 개선하는 방안을 검토하였습니다.
• 새로 접속한 경우 기존 소켓을 단순 제거하는 것이 아니라, 활성 소켓 목록에 추가하는 방식으로 변경하여 여러 기기에서 동시에 메시지를 수신할 수 있도록 설계하였습니다.
• 또한 disconnect 이벤트 발생 시 해당 소켓 ID만 배열에서 제거하도록 처리하여, 특정 기기 연결 종료가 전체 사용자 연결 종료로 이어지지 않도록 구조를 수정하였습니다.
• 이를 통해 웹소켓의 “연결 단위는 사용자(User)가 아니라 소켓(Connection)”라는 개념을 명확히 이해하게 되었습니다.

결과

문제. S3 이미지 출력 오류 (배포 환경)
문제 상황
• 이미지 업로드는 정상적으로 이루어졌으나, 배포 환경에서 S3 이미지가 출력되지 않았습니다.
• 로컬에서는 정상 동작하였으나, 배포 환경에서는 403 오류가 발생하였습니다.

실행 과정
• 먼저 파일 자체 문제인지, 접근 경로 문제인지 분리하기 위해 S3 콘솔에서 직접 접근 가능한 URL을 테스트하였습니다.
• 직접 URL은 정상 출력되었기 때문에, 문제는 "파일"이 아니라 "접근 URL 생성 로직"에 있다고 판단하였습니다.
• 프로젝트 내부에서 수동으로 조합한 URL과 AWS SDK가 생성하는 표준 URL 구조를 비교하였고, 배포 환경에서 리전 정보가 포함되지 않아 403 오류가 발생한 것을 확인하였습니다.
• 초기에는 문자열 조합 방식으로 URL을 생성하였으나, AWS SDK에서 제공하는 getUrl() 기반으로 통일하여 환경 차이로 인한 오류를 방지하였습니다.
• 향후 CloudFront 도입 등 환경 확장을 고려하여 URL 생성 방식의 일관성을 유지하는 방향으로 개선하였습니다.
• 또한 403 오류의 원인이 CORS 정책일 가능성을 고려하여, 클라이언트 직접 접근이 필요한 구조이므로 S3 버킷 정책을 점검하였습니다.
• 서버를 경유하여 이미지를 반환하는 방식도 고려하였으나, 트래픽 증가 및 서버 부하를 줄이기 위해 클라이언트가 직접 S3에 접근하는 구조를 유지하는 방향을 선택하였습니다.

결과
• 배포 환경에서도 이미지가 정상적으로 출력되도록 수정하였습니다.
• 단순 수정이 아닌, 확장성과 트래픽 분산을 고려한 구조 개선을 경험하였습니다.

• 인증 성공 시 반환되는 ID Token을 LocalStorage에 저장하는 대신, XSS 공격을 방지하기 위해 HttpOnly Cookie에 저장하는 방식을 선택하였습니다.
• HttpOnly Cookie는 JavaScript에서 접근이 불가능하기 때문에 클라이언트 스크립트 탈취 시에도 토큰이 노출되지 않는 장점이 있습니다.
• 세션 기반 인증 대신 Stateless 구조를 채택하여, 서버 확장 시 세션 동기화 부담을 줄이는 구조로 설계하였습니다.
• 이메일 인증 코드는 일시적인 데이터이기 때문에, 영속 DB 대신 Redis에 TTL 기반으로 저장하였습니다.
• 인증 만료 시간은 초 단위 TTL로 설정하여 자동 만료되도록 구성하였습니다.
• 그 이유는 이메일 인증 정보는 일시적인 데이터이기 때문에 영속 DB보다 인메모리 캐시가 적합하다고 판단했기 때문입니다.
• Docker 환경에서 단일 Redis 서버로 구성하여 서비스와 캐시 서버를 논리적으로 분리하였습니다.

결과
• 보안성과 확장성을 고려한 인증 구조를 설계하였습니다.
• 단순 로그인 구현이 아닌, 인증 아키텍처 설계 경험을 쌓을 수 있었습니다.

1. HTTPOnly Cookie

2. 스케줄러
• 경매 사이트를 개발하며 경매 마감 시간이 지나면 자동으로 입찰 및 낙찰 처리, 경매 상태 관리를 해야 한다는 필요성을 느꼈고, 이를 해결하기 위해 Spring의 스케줄러를 활용하는 방법을 배웠습니다.
• @Scheduled 어노테이션의 fixedRate 옵션을 사용하여 경매 종료 여부를 주기적으로 검사하고 상태를 업데이트하도록 구현하였으며, 이를 통해 사용자의 개입 없이 시스템이 자동으로 경매 상태를 관리할 수 있도록 했습니다.
• 스케줄러 실행 도중 오류가 발생할 경우 시스템에 영향을 주지 않도록 예외 처리를 추가하는 과정에서, 분산 환경에서 스케줄러를 안정적으로 운영하는 방법에 대한 고민을 하게 되었습니다.

3. HTTPS 배포를 위한 도메인 및 로드밸런싱
• AWS Route 53을 이용하여 도메인을 설정하고, ACM(AWS Certificate Manager)에서 SSL/TLS 인증서를 적용하는 과정을 학습하였습니다. 이를 통해 HTTPS 환경에서 안전한 웹사이트를 배포할 수 있었습니다.
• HTTPS 환경에서 로드 밸런서를 통해 트래픽을 분산하는 과정에서, HTTP → HTTPS 리디렉션 설정과 보안 그룹 설정을 조정하는 방법을 배웠습니다.이를 통해 사용자가 항상 보안 연결을 통해 서비스에 접근하도록 보장할 수 있었습니다.
• HTTPS 설정 과정에서 SSL 인증서 적용 후 발생하는 Mixed Content 문제 및 CORS 이슈를 해결하는 과정에서, 웹 애플리케이션의 보안 및 브라우저 정책에 대한 이해도를 높일 수 있었습니다.

• 이에 따라, 보다 안정적인 운영을 위해 일반적으로 t3.small(2GB RAM) 이상의 인스턴스를 사용하게 됨을 이해하였습니다.

문제 1. 로그인 정보를 세션에 저장함에 따른 확장성 및 보안 문제
문제 상황
• Firebase 로그인 성공 후 반환된 토큰을 서버 세션에 저장하여 인증 상태를 유지하였습니다.
• 이 방식은 서버 메모리에 사용자 인증 상태를 저장하는 Stateful 구조였습니다.

실행 과정
• 세션 기반 인증은 서버가 사용자 상태를 직접 보관하는 구조이므로, 서버 인스턴스가 여러 대로 확장될 경우 세션 공유 문제가 발생할 수 있습니다.
• 또한 Firebase는 이미 서명된 ID 토큰을 발급해주므로, 서버가 인증 상태를 다시 저장하는 것은 중복 관리라고 판단하였습니다.
• 따라서 서버는 상태를 저장하지 않고, 요청마다 Firebase 토큰을 검증하는 Stateless 구조가 더 적합하다고 판단하였습니다.
• 토큰 저장 위치로 LocalStorage 대신 HttpOnly Cookie를 선택한 이유는, XSS 공격으로부터 토큰 탈취를 방지하기 위함이었습니다.
• HttpOnly + Secure 옵션을 적용하여 클라이언트 스크립트 접근을 차단하고 HTTPS 환경에서만 전송되도록 구성하였습니다.

결과
• Stateful 세션 구조를 제거하고 Stateless 인증 구조로 개선하였습니다.
• 서버 확장성과 보안성을 동시에 확보하였습니다.
문제 2. 로그인 정보를 세션에 저장함에 따른 구조적 한계
문제 상황
• Firebase 로그인 성공 후 반환된 토큰을 서버 세션에 저장하여 인증 상태를 유지하였습니다.
• 서버가 사용자 상태를 보관하는 Stateful 구조였습니다.

실행 과정
• 인증 유지 방식으로 세션 유지, LocalStorage 저장, HttpOnly Cookie 저장 세 가지 방식을 비교하였습니다.
• 세션 유지 방식은 서버 확장 시 세션 공유(Redis 세션 클러스터링 등)가 필요하여 구조가 복잡해진다는 단점이 있었습니다.
• LocalStorage 저장 방식은 구현은 단순하지만, JavaScript에서 접근 가능하므로 XSS 공격에 취약하다는 문제가 있었습니다.
• 반면 HttpOnly Cookie는 클라이언트 스크립트 접근이 불가능하여 토큰 탈취 위험을 줄일 수 있었습니다.
• 또한 Firebase가 이미 서명된 ID 토큰을 발급하므로, 서버는 상태를 저장하지 않고 요청마다 검증하는 Stateless 구조가 더 적합하다고 판단하였습니다.
• 따라서 세션 기반 구조를 제거하고, HttpOnly + Secure Cookie 기반의 Stateless 인증 방식을 선택하였습니다.

결과
• 보안성과 확장성을 동시에 고려한 인증 구조로 개선하였습니다.
문제 3. AWS 배포 시 Secure 쿠키 전송 실패
문제 상황
• Secure 옵션이 설정된 쿠키가 HTTP 환경에서 전송되지 않았습니다.

실행 과정
• 해결 방법으로는 Secure 옵션 제거 또는 HTTPS 환경 구축 두 가지가 있었습니다.
• Secure 옵션을 제거하면 즉시 문제는 해결되지만, 인증 토큰이 평문 HTTP로 전송되어 보안 수준이 낮아지는 문제가 있었습니다.
• 인증 토큰은 사용자 권한과 직결되는 정보이므로, 개발 편의성보다 보안을 우선적으로 고려해야 한다고 판단하였습니다.
• EC2에 직접 SSL을 적용하는 방식과 AWS Load Balancer에서 TLS를 종료하는 방식을 비교하였습니다.
• 향후 서버 확장 가능성과 인증서 중앙 관리의 용이성을 고려하여 Load Balancer + ACM 구조를 선택하였습니다.
• 결과적으로 HTTPS 환경을 구축하여 Secure 옵션을 유지하였습니다.

결과
• 인증 쿠키가 정상적으로 전송되며 로그인 상태가 안정적으로 유지되었습니다.
• 인증 토큰이 암호화된 HTTPS 통신을 통해 전달되도록 개선하여 보안 수준을 실서비스 환경에 맞게 강화하였습니다.
• 향후 서버 인스턴스가 추가되더라도 Load Balancer 구조에서 동일한 인증 환경을 유지할 수 있도록 확장성을 확보하였습니다.
문제 4. 이메일 인증 코드 저장 방식 설계
문제 상황
• 이메일 인증 코드는 일정 시간이 지나면 자동 만료되어야 하는 일회성 데이터였습니다.

문제 1. MQTT 통신 테스트 환경 부재
문제 상황
• 외부 회사가 Publish하고 우리가 Subscribe하는 구조였기 때문에, 실제 운영 환경에 연결하기 전까지 자체적인 검증이 어려웠습니다.

실행 과정
• 테스트 방법으로는 ① 코드 내부 Mock 데이터 주입, ② 실제 운영 서버에 직접 연결 요청, ③ 로컬 브로커 구축 세 가지를 고려하였습니다.
• Mock 방식은 구현은 가장 단순하지만, MQTT는 네트워크 연결 상태, 브로커 세션, JSON 직렬화/역직렬화 과정 등 실제 통신 레벨에서 발생하는 문제를 검증할 수 없다는 한계가 있었습니다.
• 운영 서버에 직접 테스트를 요청하는 방식은 협업 의존도가 높고, 준비되지 않은 상태에서 반복 테스트를 요청하는 것은 프로젝트 리스크를 증가시킨다고 판단하였습니다.
• 따라서 실제 네트워크 기반 통신 구조를 그대로 재현할 수 있는 로컬 브로커 구축 방식을 선택하였습니다.
• Mosquitto는 경량 MQTT 브로커로 설정이 단순하고, Docker 환경에서 빠르게 컨테이너로 실행 가능하다는 장점이 있었습니다.
• Docker를 사용한 이유는, 로컬 OS 환경과 독립된 네트워크 환경을 구성하여 운영 환경과 유사한 구조를 재현하고, 설정 충돌 없이 반복 테스트가 가능하도록 하기 위함이었습니다.
• CMD에서 직접 JSON 메시지를 Publish하고, 프로젝트에서 Subscribe하여 수신 및 파싱을 검증함으로써 실제 운영 연결 전 통신 구조를 완전히 준비된 상태로 만들었습니다.

결과
• 외부 의존성을 제거한 독립 테스트 환경을 확보하였습니다.
• 운영 연동 시 구조 수정 없이 안정적으로 수신에 성공하였습니다.
문제 2. MQTT 통신 구조에 대한 역할 혼선
문제 상황
• 초기에는 양측 모두가 브로커를 구축해야 한다고 이해하고 있었으며, 서로를 구독자로 인식하는 구조적 혼선이 있었습니다.

실행 과정
• 통신 구조로는 ① 각 회사가 개별 브로커 운영 후 브리징, ② 한쪽 회사가 브로커 단독 운영, ③ 중앙 브로커 단일화 세 가지 방안을 검토하였습니다.
• 개별 브로커 브리징 방식은 설정 복잡도와 장애 지점이 증가하며, 브로커 간 동기화 문제 가능성이 존재한다고 판단하였습니다.
• 한쪽 회사가 단독 브로커를 운영하는 방식은 운영 책임과 장애 대응 부담이 특정 회사에 집중되는 구조였습니다.
• 발주처 측에서 이미 운영 중인 MQTT 브로커가 존재하였기 때문에, 해당 서버를 중앙 브로커로 활용하는 단일 구조가 가장 단순하고 책임 구조가 명확하다고 판단하였습니다.
• 상대 회사는 Publish, 우리는 Subscribe하는 역할을 명확히 정의하고, TLS 인증서를 적용하여 보안 연결을 구성하였습니다.
• 이를 통해 통신 흐름을 “Publisher → 중앙 브로커 → Subscriber”로 단순화하였습니다.

결과
• 구조적 혼선을 해소하고 역할을 명확히 정리하였습니다.
• 서버 운영 부담을 최소화하면서 안정적인 통신 구조를 확보하였습니다.

문제 1. 수백 개 장비 동시 표출로 인한 가독성 및 렌더링 성능 저하
문제 상황
• VMS, LCS, CCTV 등 수백 개 이상의 장비를 카카오지도 위에 동시에 렌더링해야 했습니다.
• 모든 장비를 한 화면에 동시에 표시할 경우 마커 객체가 과도하게 증가하여 브라우저 렌더링 성능이 저하되고, 실제 상황 인지가 어려워지는 문제가 발생하였습니다.

실행 과정
• 해결 방안으로는 ① 서버에서 데이터 개수를 제한하는 방식, ② 지도 엔진을 Canvas/WebGL 기반으로 변경하는 방식, ③ 프론트 레벨에서 레이어 구조를 재설계하는 방식 세 가지를 검토하였습니다.
• 서버 제한 방식은 상황판의 본질인 “전체 장비 모니터링” 목적과 맞지 않았습니다.
• 지도 엔진을 교체하는 방식은 기술 스택 전환 비용이 크고, 기존 카카오지도 API와의 연동을 재구성해야 하는 부담이 있었습니다.
• 따라서 기존 스택을 유지하면서 구조적으로 해결할 수 있는 레이어 분리 방식을 선택하였습니다.
• 장비 유형별(VMS, LCS, CCTV)로 레이어를 분리하여 논리적 그룹을 형성하였고, 동시에 모든 레이어를 노출하지 않고 순환 표출 구조를 도입하였습니다.
• 이는 단순 성능 개선이 아니라, “사용자의 인지 부담을 줄이는 설계”라는 판단이었습니다.
• 추가적으로 카카오지도 기본 줌 레벨 기반 클러스터링을 활용하여 확대 수준에 따라 마커 수를 동적으로 줄임으로써 렌더링 객체 수를 최소화하였습니다.
• React 상태 변경 시 전체 리렌더링이 발생하지 않도록 레이어 단위로 상태를 관리하여 불필요한 DOM 업데이트를 줄였습니다.

결과
• 수백 개 장비를 유지하면서도 렌더링 부하를 안정화하였습니다.
• 정보 과밀 문제를 해결하여 상황 인지 효율을 개선하였습니다.
문제 2. 공공 API 호출 제한 및 구조적 비효율 문제
문제 상황
• 초기에는 프론트 또는 서버에서 외부 지하철 실시간 API를 직접 호출하여 화면에 표출하는 구조였습니다.
• 일일 호출 제한으로 인해 API 차단이 발생하였으며, 다수 사용자가 접속할 경우 호출 수가 기하급수적으로 증가하는 문제가 있었습니다.

실행 과정
• 해결 방안으로는 ① 프론트 호출 최적화, ② Redis 캐시 도입, ③ 서버 단일 수집 후 DB 업데이트 구조 세 가지를 검토하였습니다.
• 프론트 호출 최적화는 사용자 수 증가 시 근본적인 해결책이 될 수 없었습니다.
• Redis 캐시를 도입하는 방식도 고려하였으나, 외부 API 호출 자체가 줄어들지 않으면 일일 제한 문제는 여전히 존재한다고 판단하였습니다.
• 따라서 API 호출을 백엔드 단일 지점에서 통제하고, 10초 주기로 데이터를 수집하여 DB를 업데이트하는 구조로 변경하였습니다.
• 특히 지하철역과 열차는 고정된 엔티티이므로 무한 적재(Insert)가 아닌 Update 방식이 저장 공간과 조회 성능 측면에서 효율적이라고 판단하였습니다.
• 프론트는 외부 API가 아닌 DB 조회 방식으로 전환하여 사용자 수와 무관하게 안정적인 데이터 제공이 가능하도록 설계하였습니다.
• 이는 “실시간성 극대화”가 아니라 “운영 안정성 확보”를 우선한 구조적 선택이었습니다.

결과
• API 호출 제한 문제를 근본적으로 해결하였습니다.
• 사용자 수 증가와 무관한 안정적인 데이터 구조를 확보하였습니다.
문제 3. 잦은 요구사항 변경 속 설계 일관성 유지 문제
문제 상황
• ITS 상황판 특성상 발주처 및 유관기관의 요구사항 변경이 빈번하였습니다.
• 단순 UI 수정이 아닌, 데이터 의미와 표출 기준 변경 요청도 포함되어 있었습니다.

실행 과정
• 변경 요청을 즉각 반영하는 방식은 단기적으로는 빠른 대응처럼 보일 수 있으나, 전체 설계 일관성을 무너뜨릴 위험이 있다고 판단하였습니다.
• 특히 상황판은 단순 시각화가 아니라 “의사결정을 지원하는 정보 체계”이기 때문에 데이터의 의미와 목적이 명확해야 한다고 보았습니다.
• 따라서 변경 요청 발생 시 해당 데이터가 실제 의사결정에 어떤 의미를 가지는지부터 재정의하였습니다.
• 기존 DB 구조는 기구축 상태였기 때문에 테이블 구조를 변경하기보다는 기존 모델을 유지한 채 확장하는 방향을 선택하였습니다.
• UI 레벨 변경은 컴포넌트 단위로 분리하여 핵심 데이터 로직과 분리하는 구조를 유지하였습니다.
• 이는 기능 추가가 반복되더라도 핵심 도메인 구조가 흔들리지 않도록 하기 위한 설계적 판단이었습니다.

결과
• 잦은 요구사항 변경에도 불구하고 전체 구조의 일관성을 유지하였습니다.
• 데이터 의미 중심의 설계를 유지함으로써 상황판의 목적을 훼손하지 않았습니다.
