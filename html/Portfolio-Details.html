문제 1. 지역 코드 불일치로 인한 지도 시각화 오류
문제 상황
• 지도 시각화 결과 일부 지역만 표시되지 않는 현상을 발견했습니다. 분석 데이터는 정상적으로 존재했기 때문에 데이터 누락이 아닌 지도와 데이터 간 매칭 문제라고 판단했습니다.
해결 과정
• 원인을 파악하기 위해 GeoJson 파일과 엑셀 데이터의 구조를 비교했습니다. 그 결과, 엑셀 데이터의 지역 코드와 GeoJson 파일의 지역 키 형식이 서로 다르다는 점을 확인했습니다.

• 해결 방안으로는 GeoJson 파일의 키 구조를 수정하는 방법과 엑셀 데이터의 지역 코드를 GeoJson 형식에 맞게 수정하는 방법 두 가지를 고려했습니다.

• GeoJson 파일은 지도 전체 구조에 직접적인 영향을 주는 핵심 데이터였기 때문에, 구조 변경 시 발생할 수 있는 시각화 오류를 최소화하고자 엑셀 데이터의 지역 코드를 수정하는 방법을 선택했습니다.
결과 및 개선
• 데이터 키가 정상적으로 매칭되면서 누락되었던 지역이 지도에 정상적으로 표시되었고, 추가적인 시각화 오류 없이 문제를 해결할 수 있었습니다.
문제 2. GeoJson 지역 누락으로 인한 시각화 미표시 문제
문제 상황
• 기존 GeoJson 파일을 수정하여 사용하던 중, 분석 대상 지역의 데이터는 존재했지만 지도에는 해당 지역이 표시되지 않는 현상을 발견했습니다. 이로 인해 GeoJson 데이터 자체에 지역 누락이 발생했음을 인지했습니다.
해결 과정
• 문제 해결을 위해 두 가지 방법을 검토했습니다. 첫째, 기존 GeoJson 파일에 누락된 지역을 직접 추가하는 방법과 둘째, 분석 대상과 구조가 일치하는 새로운 GeoJson 파일을 적용하는 방법이었습니다.

• 기존 파일을 직접 수정하는 경우, 좌표 오류나 지도 형태 왜곡이 발생할 가능성이 있다고 판단했습니다. 이에 따라 전국을 8도로 구분한 GeoJson 파일을 새로 탐색하여 적용했고, 분석 데이터와 정확히 일치하는 구조로 변경하는 방식을 선택했습니다.
결과
• 누락되었던 지역이 지도에 정상적으로 표시되었으며, 의도한 전국 8도 형태의 지도 시각화를 안정적으로 구현할 수 있었습니다.

• 지속적인 연결을 유지하기 때문에 실시간성이 중요한 서비스에서 웹소켓이 더 적합합니다.

• HTTPS 설정 과정에서 SSL 인증서 적용 후 발생하는 Mixed Content 문제 및 CORS 이슈를 해결하는 과정에서, 웹 애플리케이션의 보안 및 브라우저 정책에 대한 이해도를 높일 수 있었습니다.

• SMTP 서버에서 발생하는 보안 이슈(2단계 인증, 앱 비밀번호 설정 등)를 해결하는 과정에서, 외부 API와 연동할 때의 보안 고려 사항과 설정 방법을 익힐 수 있었습니다.

5. HTTPS 배포 시 t3.small 이상의 인스턴스를 사용하는 이유
• HTTPS(SSL/TLS) 암호화 및 복호화 과정에서 높은 RAM 사용량이 발생함을 학습하였습니다.
• 로그인 이후 HttpOnly 쿠키를 지속적으로 포함하여 처리하는 과정에서 추가적인 리소스가 요구됨을 경험하였습니다.
• 기본적으로 t2.micro 인스턴스는 1GB RAM을 제공하며, Spring Boot와 HTTPS 처리까지 수행하기에는 부족함을 확인하였습니다.
• 이에 따라, 보다 안정적인 운영을 위해 일반적으로 t3.small(2GB RAM) 이상의 인스턴스를 사용하게 됨을 이해하였습니다.

1. 로그인정보를 세션에 저장함에 따라 보안성이 취약함

• 일정 시간이 지나면 경매 상태를 '마감'으로 바꾼 후, 낙찰을 수동으로 진행하였습니다.
• 하지만, 경매 물품이 많아짐에 따라 경매 상태 및 낙찰을 수동을 관리할 수는 없게 되었습니다.

3. 별도 설정 없이 AWS에서 배포할 경우 HTTPOnly 쿠키가 정상적으로 동작하지 않는 문제
• AWS에서 별도의 설정없이 배포하면 HTTP 프로토콜로 배포되었습니다.

4. 배포 시 SMTP 이메일 발송 오류 문제
• 도커 이미지 빌드 후 EC2에 업로드 시, 이메일이 발송되지 않았습니다.
해결법
1. HTTPOnly 쿠키를 통한 보안성 강화
• 로그인 정보를 HTTPOnly 쿠키에 저장하여, 브라우저에서 쿠키에 직접 접근할 수 없도록 설정하였습니다.이를 통해 보안성이 강화되고 XSS 공격으로부터 정보를 보호할 수 있었습니다.
• 기존에는 세션에 로그인 정보를 저장했지만, 이를 HTTPOnly 쿠키로 변경하면서 인증 방식이 더 안전하고 간편해졌습니다.
• 배포 환경에서도 HTTPOnly 쿠키가 정상적으로 동작하도록 설정하는 과정에서, HTTPS 환경이 필요하다는 것을 알게 되었습니다.

2. 스케줄러를 활용한 경매 상태 자동 관리
• Spring의 @Scheduled 어노테이션을 사용하여 0.6초마다 경매 상태를 검사하도 구현하였습니다. 이를 통해 일정 시간이 지나면 자동으로 경매 상태를 '마감'으로 변경할 수 있었습니다.
• 경매 마감 시 자동으로 낙찰 처리를 수행하도록 로직을 구성하여, 사용자의 개입 없이 시스템이 입찰자 중 최고가를 제시한 사용자를 낙찰자로 지정하도록 개선하였습니다.
• 경매 물품 수가 증가해도 안정적으로 동작하도록, 스케줄러 실행 시 불필요한 데이터 조회를 최소화하고 성능을 최적화하는 방법을 적용하였습니다.

3. HTTPS 배포를 위한 도메인 및 로드 밸런서 설정
• Route 53과 가비아를 이용해 도메인을 구매하고, AWS ACM에서 SSL 인증서를 적용하여 HTTPS 환경을 구축하였습니다.
• AWS 로드 밸런서를 설정하고, 타겟 그룹을 구성하여 트래픽을 관리하도록 개선하였습니다. 이를 통해 사용자가 항상 HTTPS 프로토콜을 통해 안전하게 접속할 수 있도록 설정하였습니다.
• HTTP → HTTPS 자동 리디렉션을 적용하여, 보안성이 낮은 HTTP 연결을 차단하고 모든 요청이 보안 연결을 통해 이루어지도록 구성하였습니다.

4. 도커 컨테이너를 활용한 이메일 인증 문제 해결
• 도커 컨테이너 빌드 시, 애플리케이션을 관리하는 컨테이너와 Redis 컨테이너를 분리하여 두 개의 컨테이너를 생성하였습니다. 이를 통해 이메일 인증 요청 시 Redis를 활용한 비동기 처리가 가능하도록 개선하였습니다.
• EC2 환경에서 두 컨테이너가 정상적으로 연결되도록 Docker Compose를 사용하여 네트워크 설정을 조정하였으며, 이메일 전송 시 SMTP 서버와의 연결이 원활하게 이루어지도록 환경 변수를 관리하였습니다.
• 컨테이너 내부에서 SMTP 인증이 정상적으로 동작하지 않는 문제를 해결하기 위해, 필요한 인증 정보를 컨테이너 실행 시 환경 변수로 주입하고, SMTP 포트가 차단되지 않도록 보안 그룹을 조정하였습니다.

• MQTT의 발행(Publish)–구독(Subscribe) 구조를 이해하며, 네트워크가 불안정할 수 있는 환경에서도 실시간 데이터를 효율적으로 전달하는 무선 통신 방식에 대한 전반적인 이해를 넓힐 수 있었습니다.
• 실제 적용 전 셀프 테스트 과정에서 Docker를 활용해 MQTT 환경을 구성하며, 개발 환경을 분리하고 반복 테스트를 진행하는 과정에서 컨테이너 기반 개발 방식에 대한 이해도 함께 높일 수 있었습니다.

2. FCM 토큰 기반 앱 푸시 알림 구조 이해
• 돌발 상황 발생 시 즉각적인 대응을 위해, FCM(Firebase Cloud Messaging) 토큰을 이용한 앱 푸시 알림 방식에 대해 학습하였습니다.
• 사용자별 FCM 토큰을 기반으로 알림을 전송하는 구조를 이해하며, 이벤트 발생 시 특정 대상에게만 알림을 전달할 수 있는 앱 푸시 알림 시스템의 동작 원리를 파악할 수 있었습니다.
• 이를 통해 웹 화면 중심의 알림을 넘어, 모바일 앱 푸시 알림이 실시간 상황 인지와 대응 속도에 중요한 역할을 한다는 점을 배울 수 있었습니다.

3. GIS 좌표계에 대한 인식과 중요성 이해
• 프로젝트를 진행하며 GIS에는 다양한 좌표계가 존재하며, 동일한 위치 데이터라도 좌표계에 따라 지도 상 위치가 크게 달라질 수 있다는 사실을 처음으로 인지하게 되었습니다.
• 단순히 위·경도를 사용하는 것만으로는 정확한 공간 표현이 어렵고, 서비스 전반에서 기준 좌표계를 명확히 정의하고 통일하는 것이 필수적이라는 점을 배웠습니다.
• 이를 통해 GIS 서비스의 정확성과 신뢰성은 좌표계에 대한 이해에서부터 시작된다는 점을 체감하게 되었습니다.

문제점
1. MQTT 기반 실시간 데이터 테스트의 어려움
• MQTT 통신은 다른 회사에서 데이터를 송신하고, 우리 회사에서 이를 수신하는 구조였기 때문에 단독으로 테스트 환경을 구성하는 데 어려움이 많았습니다.
• 실제 운영 환경과 동일한 조건을 재현하기 힘들어, 데이터 수신 여부와 메시지 흐름을 검증하는 과정에서 시행착오를 겪었습니다.

2. MQTT 통신 구조에 대한 소통 오류 문제
• 프로젝트 초기에 양측 모두가 서로를 구독자로 인식하고 있었으며, 각각 서버를 구축하는 역할로 이해하고 있어 통신 구조에 혼선이 발생하였습니다.
• 이로 인해 Publish–Subscribe 구조가 명확히 정리되지 않았고, 실제 데이터 흐름을 정의하고 조율하는 과정이 쉽지 않았습니다.

3. 히트맵 시각화 과정에서의 표현 오류 문제
• 관객 밀집도를 보다 자연스럽게 표현하기 위해 히트맵의 부드러운 움직임을 구현하고자 하였습니다.
• 하지만 시각적 효과를 강화하는 과정에서 오히려 데이터가 왜곡되어 보이는 문제가 발생하였고, 실제 관객 분포와 다르게 인식될 수 있다는 한계를 경험하였습니다.

해결법
1. Docker 기반 테스트 환경 구축을 통한 MQTT 통신 검증
• 상대 회사로부터 사전에 전달받은 데이터 형식을 기반으로, Docker Desktop을 활용해 MQTT 테스트 환경을 구성하였습니다.
• 테스트용 데이터를 직접 발행(Publish)하여 프로젝트에서 정상적으로 수신 및 파싱되는지를 반복적으로 검증하였습니다.
• 테스트를 통해 데이터 수신 구조를 사전에 안정화한 후, 상대 회사에 검증 완료를 공유하였고 실제 운영 데이터 발신 시에도 문제없이 수신 및 처리할 수 있었습니다.

2. 발주처 서버를 활용한 MQTT 중계 구조로 통신 문제 해결
• 시간 제약으로 인해 상대 회사와 우리 회사 모두 별도의 서버를 구축하기 어려운 상황이었습니다.
• 다행히 발주처 측에서 이미 구축된 서버를 보유하고 있어, 해당 서버를 MQTT 통신의 중간 브로커 역할로 활용하는 구조를 협의하였습니다.
• 상대 회사는 발주처 서버로 데이터를 송신하고, 우리 회사는 발주처 서버에 구독 방식으로 연결하여 데이터를 수신함으로써 통신 구조를 단순화하고 안정적인 데이터 수신에 성공하였습니다.

3. 히트맵 시각화 로직 개선을 통한 정확도 향상
• 히트맵이 일그러져 보이던 원인이 부드러운 움직임을 위한 가중치 처리 방식에 있음을 파악하였습니다.
• 가중치 기반 보정 대신, 데이터 수신 주기와 시각화 처리 간의 시간 지연을 최소화하는 방향으로 로직을 수정하였습니다.
• 이를 통해 히트맵의 자연스러운 움직임을 유지하면서도, 실제 데이터에 기반한 정확한 관객 밀집도 시각화에 성공하였습니다.

배운 내용
1. 근무자 중심의 데이터 선별 및 상황판 표출 방식 설계
• 상황판은 단순히 보기 쉽게 구성하는 것이 아니라, 근무자가 한눈에 인지했을 때 의미를 해석할 수 있는 정보를 제공하는 것이 중요하다는 점을 배웠습니다.
• 기본적인 도메인 지식을 보유한 사용자가 사용하는 시스템인 만큼, 사용자 조작을 과도하게 제한하기보다 패널에 표출되는 데이터 자체의 구성과 맥락에 집중하는 설계 방식이 필요하다는 점을 이해했습니다.
• 이를 통해 상황판 UI는 편의성보다 정보 전달력과 즉시성이 우선되어야 한다는 관점을 갖게 되었습니다.

2. 설계 감리 참여를 통한 문서 설계의 중요성에 대한 이해
• 설계 감리에 직접 참여하며, 설계 문서가 단순한 참고 자료가 아니라 사업 전반을 규정하는 기준 문서라는 점을 깊이 이해하게 되었습니다.
• 설계 단계에서 문서가 명확하지 않거나 불완전할 경우, 종료 감리 단계에서 업무 범위가 크게 증가하거나 현실적으로 구현이 어려운 기능까지 요구되는 상황이 발생할 수 있음을 경험했습니다.
• 특히 설계 감리에서 걸러지지 않은 기능은 이후 구현 단계에서 불필요하거나 불가능한 기능임에도 불구하고 이를 맞추기 위해 과도한 개발 리소스를 투입해야 하는 문제로 이어질 수 있다는 점을 배웠습니다.
• 이 과정을 통해 설계 문서를 어떻게 작성해야 하는지뿐 아니라, 왜 설계 문서를 정확하고 책임 있게 작성해야 하는지를 체감하게 되었으며, 설계 감리를 통해 프로젝트 단위가 아닌 사업 전반의 흐름과 구조를 이해할 수 있는 시야를 갖게 되었습니다.

3. 대규모 지도 렌더링 환경에서의 성능 최적화 경험
• 다수의 장비와 다양한 종류의 데이터를 한 화면에 동시에 표출할 경우 가독성과 성능 모두에 한계가 발생한다는 문제를 경험했습니다.
• 이를 해결하기 위해 레이어 단위로 정보를 순환 표출하고, 클러스터링 기법을 적용해 지도 렌더링 부하를 줄이는 방식을 적용했습니다.
• 해당 경험을 통해 GIS 상황판에서는 정보의 양을 줄이는 설계 자체가 성능 최적화가 될 수 있다는 점을 배웠습니다.

문제점
1. 다수 장비 동시 표출로 인한 상황판 가독성 및 성능 저하
• ITS 상황판 특성상 CCTV, 라이다, 센서 등 다양한 장비 정보를 동시에 표출해야 했습니다.
• 모든 장비를 한 화면에 동시에 렌더링할 경우 화면이 과도하게 복잡해지고, 브라우저 렌더링 부하로 인해 실제 상황 인지가 어려운 문제가 발생하였습니다.

2. 지하철 실시간 API 호출 제한 및 캐시 문제
• 지하철 실시간 정보 수집을 위해 외부 공공 API를 활용하였으나, 일일 호출 횟수 제한으로 인해 최대 1,000건도 채우지 못하고 API가 차단되는 문제가 발생하였습니다.
• 또한 실시간 API를 그대로 호출하여 화면에 표출하는 구조로 인해 중복 호출과 캐시 문제가 함께 발생하였습니다.

3. 잦은 요구사항 변경으로 인한 설계 일관성 저하 우려
• ITS 상황판 사업 특성상 발주처 및 유관기관의 요구사항 변경이 빈번하게 발생하였습니다.
• 단순 UI 변경을 넘어 데이터 표출 기준이나 정보 구성 방식에 영향을 주는 요청도 포함되어 있어, 무분별한 반영 시 전체 설계 구조가 흔들릴 위험이 있었습니다.

해결법
1. 장비별 레이어 분리 및 순환 표출 방식 도입
• 장비 유형별로 레이어를 분리하여 동시에 모든 정보를 표출하지 않도록 설계하였습니다.
• 화면에서는 일정 주기로 레이어를 순환 표출하도록 구성하여, 가독성을 유지하면서도 전체 정보를 단계적으로 확인할 수 있도록 개선하였습니다.
• 또한 클러스터링 기법을 적용하여 렌더링 부하를 줄이고 성능을 안정화하였습니다.

2. API 호출 구조 개선 및 데이터 적재 방식 전환
• 서울 열린데이터 광장에 공식적으로 API 키 사용 제한 해제를 요청하여 일일 호출 제한 문제를 우선적으로 해결하였습니다.
• 실시간 API를 직접 호출하여 표출하는 방식 대신, 백엔드에서 API 데이터를 주기적으로 수집하여 DB에 업데이트하는 구조로 변경하였습니다.
• 화면에서는 DB 데이터를 조회하는 방식으로 전환하여 캐시 문제를 해소하고 안정적인 데이터 제공이 가능하도록 개선하였습니다.

3. 단계적 협의 기반 요구사항 반영 및 핵심 정보 중심 설계 유지
• 요구사항 변경 발생 시 즉각적인 반영이 아닌, 여러 차례 회의를 통해 변경 목적과 필요성을 명확히 정리하였습니다.
• 전체 설계를 크게 변경하기보다는 기존 구조를 최대한 유지하는 방향으로 수정 범위를 제한하였습니다.
• 상황판에서 반드시 유지되어야 할 핵심 정보와 지표를 기준으로 의미 있는 정보는 보존하고 불필요한 변경은 최소화하여 설계 일관성을 유지하였습니다.
